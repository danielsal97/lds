🎯 Introduction & Project Overview

What Are We Building?

LDS is a distributed storage system that combines:

  • 📱 IoT Devices (Raspberry Pis) as storage nodes
  • 🔌 One Master Server that coordinates everything
  • 💾 RAID01 fault tolerance (data replicated across minions)
  • 🔌 Plugin System for dynamic extensions
  • 📁 Linux Filesystem Interface (NBD) for easy access
Think of it like this: Instead of buying an expensive NAS box, you can use cheap Raspberry Pis as storage nodes, and LDS coordinates everything. You get the reliability of RAID01 (data survives if one device fails) without needing specialized hardware.

Key Benefits

  • ✅ Cost-effective - Use cheap IoT devices
  • ✅ Scalable - Add more minions anytime
  • ✅ Reliable - RAID01 protects your data
  • ✅ Extensible - Plugins for custom functionality
  • ✅ User-friendly - Works like a normal Linux directory

🏗️ What is LDS?

The Big Picture

LDS creates a virtual storage device that your Linux system sees as a regular block device. Behind the scenes, data is distributed and replicated across multiple IoT devices.

Three Main Roles

🖥️ Master Server

Runs LDS. Makes decisions about where data goes, coordinates all operations.

Key Tasks:
  • Receive READ/WRITE requests
  • Manage RAID01 replication
  • Track minion status
  • Handle failures

📱 Minion Devices

IoT devices (Raspberry Pis) that actually store the data. Dumb devices that just follow orders.

Key Tasks:
  • Store data blocks
  • Retrieve blocks on request
  • Report health status
  • Execute minion-side commands
Why This Design?

Separating master and minions allows the master to be powerful (make complex decisions) while minions can be cheap and simple. The master handles all the hard work.

Why This Complexity?

Component by Component:
  • Reactor: We can't block waiting for I/O. Reactor handles thousands of requests without blocking threads.
  • Input Mediator: Decouples the fast Reactor from potentially slow ThreadPool processing.
  • ThreadPool: Commands run in parallel on multiple threads for better throughput.
  • WPQ (Priority Queue): Admin commands run before low-priority ones.
  • RAID01 Manager: Decides which minions store which blocks (fault tolerance).
  • MinionProxy: Handles network communication with dumb minion devices.
  • ResponseManager: Minions are slow - we need async response handling with retries.
  • Plugin System: Extend behavior without recompiling.

👑 Master-Minion Communication Model

The Master is the Coordinator

The master makes ALL decisions. Minions are simple and just follow orders:

Master Responsibilities

  • Receive user requests
  • Decide which minions to use
  • Manage data replication
  • Handle minion failures
  • Track metadata
  • Make consistency decisions

Minion Responsibilities

  • Store blocks of data
  • Retrieve blocks on request
  • Perform simple operations
  • Report health status
  • Follow master orders
  • Never make decisions

Communication Flow: A Simple Read

1. User Request: Application asks to read block 5
2. Master Receives: NBD → Reactor catches the request
3. Master Decides: RAID01 Manager finds block 5 is on minion 1 and minion 2
4. Master Sends: "Minion 1, send me block 5" (UDP packet)
5. Minion Responds: Minion 1 sends data back (UDP packet)
6. Master Replies: Returns data to user via NBD

Why Async Communication?

Minions are on a network - they might be slow or unreachable. We can't block the master waiting for a response. Instead:

  • Master sends request and moves on
  • ResponseManager listens for replies
  • When reply arrives, scheduler processes it
  • If no reply after timeout, retry

Fault Tolerance with RAID01

Every block is stored on 2 different minions. So if minion 1 dies, we still have the data on minion 2.

Example: Block 5 is on minion 1 and 3. Master asks minion 1. If minion 1 doesn't respond, master retries with minion 3.

💾 RAID01 Explained

What is RAID01?

RAID01 means: RAID 0 (Striping) combined with RAID 1 (Mirroring)

Simple RAID01 Example

Original File: [Block 1] [Block 2] [Block 3] [Block 4] [Block 5] ...

Stored on Minions (2 copies of each block):

Minion 1:  [Block 1] [Block 3] [Block 5] ...
Minion 2:  [Block 1] [Block 3] [Block 5] ...

Minion 3:  [Block 2] [Block 4] [Block 6] ...
Minion 4:  [Block 2] [Block 4] [Block 6] ...

Pattern:
└─ Block 1, 2, 3, 4... are striped (spread) across minions
└─ Each block is mirrored (replicated) on 2 minions
└─ Result: Data survives if ANY ONE minion dies!

Why RAID01 Instead of Other RAID Levels?

RAID Level How It Works Survives Failure Why Not Use It
RAID 0 Data striped (spread) ❌ Any device failure = data loss Too risky - not fault tolerant
RAID 1 Data mirrored (2 copies) ✅ Survives 1 failure 50% storage waste (only 2 copies)
RAID 5 Striped with parity ✅ Survives 1 failure Complex recovery, slow write operations
RAID 01 Striped + Mirrored ✅ Survives 1 failure ✅ Simple, fast, easy to implement
Why RAID01 for LDS?

It's the simplest to understand and implement. Data is just copied to 2 minions. No complex parity calculations. Fast reads (can use either copy). Simple failure recovery.

Block Layout Example

Master's RAID01 Mapping:

Block 0: Minion A, Minion C
Block 1: Minion B, Minion D
Block 2: Minion A, Minion D
Block 3: Minion B, Minion C
...

When user reads Block 2:
  1. Master checks: Block 2 is on Minion A or Minion D
  2. Master tries Minion A first
  3. If Minion A is slow/dead → retry with Minion D
  4. Either way, user gets Block 2

⚙️ Command Processing Framework (CPF)

What is CPF?

CPF is the heart of LDS. It's the generic framework that processes all commands (READ, WRITE, FLUSH, etc.) efficiently and concurrently.

The CPF Pipeline

┌──────────────────────┐
│  Request arrives     │  (READ from block 5)
└──────────────────────┘
         │
         ▼
┌──────────────────────┐
│  REACTOR             │  Receives event from NBD
│  (epoll)             │  Non-blocking, event-driven
└──────────────────────┘
         │
         ▼
┌──────────────────────┐
│  INPUT MEDIATOR      │  Converts event to command
│                      │  Decouples fast Reactor from ThreadPool
└──────────────────────┘
         │
         ▼
┌──────────────────────┐
│  WAITABLE            │  Commands wait in priority queue
│  PRIORITY QUEUE      │  Admin commands go first
│  (WPQ)               │
└──────────────────────┘
         │
    ┌────┴────┬────────┬─────────┐
    │ Work    │ Work   │ Work    │  Threads
    │ Thread  │ Thread │ Thread  │
    │ 1       │ 2      │ 3       │
    ▼         ▼        ▼         │
┌──────────────────────┐         │
│  FACTORY             │         │  Creates concrete
│                      │         │  command handlers
└──────────────────────┘         │
    │         │        │         │
    ▼         ▼        ▼         │
 ┌────────────────────────────┐  │
 │  Command Execution         │  │  Execute command
 │  (READ, WRITE, etc.)       │  │  in thread
 └────────────────────────────┘  │
    │         │        │         │
    └────────┬┴────────┴─────────┘
             │
             ▼
    ┌─────────────────────────┐
    │  RESPONSE MANAGER       │  Send response or
    │  or SCHEDULER           │  schedule retry
    └─────────────────────────┘
             │
             ▼
    ┌─────────────────────────┐
    │  Back to user via NBD   │
    └─────────────────────────┘

Key Features of CPF

  • Non-blocking: Reactor doesn't block on I/O. Uses epoll to handle thousands of events.
  • Concurrent: Multiple worker threads process commands in parallel.
  • Priority-based: Commands in WPQ are ordered by priority.
  • Pluggable: New command types can be added via plugins.
  • Async: Requests to minions are async. ResponseManager handles replies.
  • Fault-tolerant: Scheduler can retry failed operations.

CPF in Action: A Complete Example

Event: User reads file at offset 1024, length 512 bytes Step 1 - REACTOR receives event └─ epoll_wait() returns: "file descriptor has data" Step 2 - INPUT MEDIATOR converts to command └─ Creates: ReadCommand(offset=1024, length=512, priority=Medium) Step 3 - WPQ queues command └─ Adds to priority queue: [Admin cmd, READ cmd (Medium), Admin cmd] Step 4 - WORKER THREAD processes └─ Thread 2 pops READ cmd from front of queue └─ Thread 2 calls Factory.Create("READ", params) └─ Factory returns concrete ReadCommand object Step 5 - FACTORY creates handler └─ Factory looks up: "READ" → returns factory function └─ Factory function: Create new ReadCommand with params Step 6 - Command executes └─ ReadCommand.Execute() is called └─ Inside Execute(): - RAID01 Manager finds block locations - MinionProxy sends request to minion - ResponseManager waits for reply - Scheduler retries if timeout Step 7 - Response sent back └─ Data returned to user via NBD

Why Each Component?

Component Purpose Why Needed
Reactor Receive many requests efficiently Block I/O would be slow. epoll scales to thousands of connections.
Input Mediator Convert events to commands Decouples fast I/O from slow processing. Reactor stays responsive.
WPQ Order commands by priority Admin commands must run first. Normal commands run when time permits.
ThreadPool Concurrent execution Multiple cores available. Process many commands in parallel.
Factory Create command objects New command types can be added (via plugins) without changing core code.
RAID01 Manager Manage data replication Fault tolerance. Knows where every block is stored.
MinionProxy Network communication Serialization/deserialization of messages. Hides network complexity.
ResponseManager Handle async responses Network is slow. Can't block waiting. Listen for replies asynchronously.
Scheduler Retry failed ops Minions might be temporarily unreachable. Retry with exponential backoff.

🔩 Core Components Explained

1. Reactor (Event-Driven I/O)

What: Reactor pattern using epoll for non-blocking I/O multiplexing
Who uses it: CPF uses Reactor to listen for NBD events
Why it matters: Can handle thousands of concurrent connections without threads

How Reactor Works

Traditional Blocking I/O (Slow):
  Thread 1 blocks on Read → Data arrives → Process → Thread 2 blocks on Read

Reactor with epoll (Fast):
  Reactor waits on 1000 file descriptors
  "Any of you have data?"
  ✓ FD#5 has data! FD#12 has data!
  Call handlers for FD#5 and FD#12
  ✓ FD#5 has more data! Call handler again
  All handlers return fast → No threads blocked

Reactor in Code

// Pseudo-code: How Reactor works class Reactor { void Add(int fd) { epoll_ctl(epfd, EPOLL_CTL_ADD, fd); // Register fd } void Run() { while (running) { int nfds = epoll_wait(epfd, events, MAX_EVENTS); for (int i = 0; i < nfds; i++) { handler(events[i].data.fd); // Call handler for each fd } } } }

2. ThreadPool (Concurrent Execution)

What: Pool of worker threads that execute commands from a priority queue
Who uses it: CPF dispatches work to ThreadPool
Why it matters: Utilizes multiple CPU cores for parallel processing

How ThreadPool Works

Main Thread (Reactor)
  └─ Add cmd to WPQ
      └─ Command added: ReadCommand(priority=HIGH)
         ├─ Signals: "Wake up workers!"
         │
         ├─ Worker 1: Gets cmd from WPQ → Execute()
         ├─ Worker 2: Gets cmd from WPQ → Execute()
         └─ Worker 3: Gets cmd from WPQ → Execute()

Main Thread
  └─ Add another cmd to WPQ
      └─ Worker 1: Still executing previous cmd
         Worker 2: Pops new cmd → Execute()
         Worker 3: Still executing

Result: Multiple commands running in parallel!

ThreadPool Benefits

  • Concurrency: 4-core system = up to 4 commands running simultaneously
  • Fairness: WPQ ensures no starvation. All command types eventually run.
  • Responsiveness: UI remains responsive while slow operations run in background
  • Resource Control: Limit number of threads to prevent resource exhaustion

3. Waitable Priority Queue (WPQ)

What: Thread-safe queue that orders commands by priority
Who uses it: ThreadPool stores commands here
Why it matters: Ensures important commands execute first

Priority Levels

Priority Queue (Top = Next to execute):

┌─────────────────────────────────┐
│ Admin Command (HIGHEST)         │  ← Pop first
├─────────────────────────────────┤
│ Admin Command                   │
├─────────────────────────────────┤
│ High Command (e.g., FLUSH)      │
├─────────────────────────────────┤
│ High Command                    │
├─────────────────────────────────┤
│ Medium Command (e.g., READ)     │
├─────────────────────────────────┤
│ Medium Command                  │
├─────────────────────────────────┤
│ Low Command (LOWEST)            │  ← Pop last
└─────────────────────────────────┘

If system is busy:
  • Admin commands run immediately
  • High priority runs after admins
  • Medium runs if there's time
  • Low priority only runs when quiet

4. Command Pattern (ICommand)

What: Design pattern that encapsulates a request as an object
Who uses it: All operations (READ, WRITE, FLUSH) inherit from ICommand
Why it matters: Allows queueing, prioritizing, and undo/redo operations

ICommand Hierarchy

ICommand (Abstract Base)
  │
  ├─ ReadCommand
  │   └─ Execute(): Fetch block from minion, return data
  │
  ├─ WriteCommand
  │   └─ Execute(): Store block on 2 minions (RAID01)
  │
  ├─ FlushCommand
  │   └─ Execute(): Ensure all writes are persistent
  │
  ├─ TrimCommand
  │   └─ Execute(): Delete blocks, free space
  │
  └─ [Plugin] CustomCommand
      └─ Execute(): Custom logic loaded from plugin

Command in Code

class ReadCommand : public ICommand { private: uint64_t offset; uint32_t length; Buffer result; public: ReadCommand(uint64_t off, uint32_t len) : ICommand(MEDIUM), offset(off), length(len) {} void Execute() override { // RAID01 Manager finds block locations result = raid01Manager.GetBlock(offset, length); // MinionProxy fetches from minions // ResponseManager handles async reply } };

5. Factory Pattern

What: Design pattern for creating objects without specifying exact classes
Who uses it: ThreadPool uses Factory to create command objects
Why it matters: New command types can be added (via plugins) without changing core

Factory in Action

1. Plugin registers command:
   Factory.Add("READ", [] { return new ReadCommand(); });
   Factory.Add("WRITE", [] { return new WriteCommand(); });

2. At runtime, when ThreadPool needs a command:
   auto cmd = Factory.Create("READ", params);
   └─ Factory looks up "READ" → finds creator function
   └─ Calls creator function → returns ReadCommand object

3. Result: Decoupled from concrete types!
   ThreadPool doesn't know about ReadCommand, WriteCommand, etc.
   Just calls Factory.Create(type) and gets whatever command type exists

6. Plugin System (PNP)

What: System for dynamically loading and executing code at runtime
Who uses it: PNP watches directory, loads .so files, plugins register with Factory
Why it matters: Extend system without recompiling - install plugin and restart

Plugin Loading Flow

1. DirMonitor watches "plugins/" directory

2. New .so file appears:
   plugins/custom_handler.so

3. SoLoader loads the plugin:
   dlopen("plugins/custom_handler.so")

4. Plugin's init function runs:
   plugin_init() {
     Factory.Add("CUSTOM_OP", CreateCustomCommand);
     Logger.Log("Custom handler loaded");
   }

5. Factory now knows about new command type:
   Factory.Create("CUSTOM_OP") → Creates CustomCommand

6. Users can now use the new operation!
   Next READ/WRITE/CUSTOM_OP → Uses updated Factory

7. RAID01 Manager

What: Manages data replication across minions
Who uses it: Commands (READ, WRITE) query RAID01 Manager for block locations
Why it matters: Implements fault tolerance - data survives minion failures

RAID01 Manager Responsibilities

  • Mapping: Knows which blocks are on which minions
  • Replication: Ensures each block is on exactly 2 minions
  • Balancing: Distributes blocks evenly across minions
  • Recovery: If minion dies, finds replica and marks original failed
  • Persistence: Saves mapping to disk so it survives restart

8. MinionProxy

What: Abstracts network communication with minions
Who uses it: Commands use MinionProxy to send/receive from minions
Why it matters: Hides protocol complexity - just serialize and send

MinionProxy Responsibilities

  • Serialization: Convert objects to bytes for network
  • Deserialization: Convert bytes back to objects
  • Addressing: Know minion IP/port
  • Sending: Send UDP packets to minion
  • Message ID: Track which request gets which response

9. ResponseManager

What: Listens for async responses from minions
Who uses it: Scheduler waits for minion replies
Why it matters: Minions are slow/unreliable - handle async responses properly

ResponseManager Flow

Timeline:
T=0ms:   Master sends "Read block 5" to Minion 1
         Minion 1 is busy...

T=10ms:  Master sends another request to Minion 2
         ResponseManager waits for replies...

T=50ms:  ✓ Minion 2 replies! ResponseManager captures response
         Scheduler processes reply

T=100ms: ✓ Minion 1 finally replies! ResponseManager captures
         Scheduler processes this too

Result: Master didn't wait for slow Minion 1
        Handled both responses asynchronously
        Multiple requests in flight simultaneously

🔄 Data Flow & Communication

Complete Flow: User Reads a File

Step 1: USER INTERACTION
  User: cat /mnt/drive/myfile.txt
  └─ Kernel: Read offset=0, length=1024

Step 2: NBD DRIVER
  Kernel → NBD → LDS Master (via GI)

Step 3: REACTOR
  Reactor.Run() is blocking on epoll_wait()
  epoll says: "File descriptor has data!"

Step 4: INPUT MEDIATOR
  Converts NBD event to ReadCommand
  ReadCommand(offset=0, length=1024, priority=MEDIUM)

Step 5: WPQ
  InputMediator pushes command to ThreadPool's WPQ
  Signals waiting worker threads: "Wake up! Work available!"

Step 6: WORKER THREAD
  Worker 1 wakes up, pops command from WPQ
  Calls: command→Execute()

Step 7: FACTORY (Inside Execute)
  RAID01Manager finds block 0-2 are on Minion A and C
  Blocks 3-4 are on Minion B and D

Step 8: MINION PROXY (Inside Execute)
  Creates message: "GET_BLOCK offset=0 length=1024"
  Sends UDP packet to Minion A
  Stores request ID: msg_id=42

Step 9: MINION PROCESSING
  Minion A receives request
  Reads blocks from local storage
  Sends response: "RESPONSE msg_id=42 data=..."

Step 10: RESPONSE MANAGER
  ResponseManager listens on UDP port
  Receives response msg_id=42
  Triggers: Scheduler.OnResponse(msg_id=42, data)

Step 11: SCHEDULER
  Checks: Is response complete?
  If yes → Mark task complete
  If no → Retry with Minion C

Step 12: SEND RESPONSE
  Data returned to user via NBD
  User gets: 1024 bytes of file content

Complete Flow: User Writes a File

Step 1: USER WRITES
  User: echo "data" > /mnt/drive/newfile.txt
  Kernel: Write offset=0, data=[bytes]

Step 2-6: SAME AS READ
  (Reactor → InputMediator → WPQ → Worker Thread)

Step 7: FACTORY + RAID01MANAGER
  RAID01Manager decides:
  Blocks 0-2: Store on (Minion A, Minion C)
  Blocks 3-4: Store on (Minion B, Minion D)
  (Replication: Each block on 2 minions = RAID01)

Step 8: MINION PROXY (2x)
  Creates 2 messages:
    - "PUT_BLOCK offset=0 data=..." → send to Minion A
    - "PUT_BLOCK offset=0 data=..." → send to Minion C
  Stores request IDs: msg_id=43, msg_id=44

Step 9: MINION PROCESSING
  Minion A receives and stores block 0
  Minion C receives and stores block 0 (same data)
  Both send responses

Step 10-11: RESPONSE MANAGER + SCHEDULER
  ResponseManager gets both responses: msg_id=43, msg_id=44
  Scheduler checks: Both minions succeeded?
  ✓ Yes → Data is safely stored on 2 copies

Step 12: SEND RESPONSE
  WRITE operation complete
  User gets: Operation succeeded

RAID01 BENEFIT:
  If Minion A dies tomorrow, block 0 still exists on Minion C!
  If Minion C dies, block 0 still exists on Minion A!
  Data survives any single minion failure.

Async Communication: Why It's Complex but Necessary

The Problem:

Minions are on a network. They might be slow, overloaded, or unreachable.

If master blocks waiting for minion, all other requests stall. That's terrible!

The Solution:

Master sends request and immediately continues processing other requests.

ResponseManager listens for replies in background.

When reply arrives, Scheduler processes it.

If minion times out, Scheduler retries with replica.

The Trade-off:

More complex (async is hard), but much better performance and reliability.

⚙️ Framework Completions & Components

1. ICommand Framework

Commands are the fundamental units of work in the CPF. Every operation (READ, WRITE, FLUSH) is a command.

Command Hierarchy

class ICommand { public: enum Priority { LOW, MED, HIGH, ADMIN }; explicit ICommand(Priority p = MED) : priority_(p) {} virtual ~ICommand() = default; virtual void Execute() = 0; // Subclasses implement Priority GetPriority() const { return priority_; } bool operator<(const ICommand& other) const { return priority_ < other.priority_; } private: Priority priority_; }; class ReadCommand : public ICommand { public: ReadCommand(uint64_t offset, uint32_t length) : ICommand(MED), offset_(offset), length_(length) {} void Execute() override; private: uint64_t offset_; uint32_t length_; }; class WriteCommand : public ICommand { public: WriteCommand(uint64_t offset, const Buffer& data) : ICommand(MED), offset_(offset), data_(data) {} void Execute() override; private: uint64_t offset_; Buffer data_; };

2. InputMediator

Bridges the fast Reactor and the potentially slow ThreadPool. Converts NBD events to Command objects.

InputMediator Purpose & Design

Why separate InputMediator from Reactor?
Reactor must stay responsive. If creating commands is slow, Reactor blocks. InputMediator decouples them - Reactor quickly delegates to InputMediator, then gets back to listening for more events.
class InputMediator { public: explicit InputMediator(ThreadPool& pool) : thread_pool_(pool) {} void OnNBDEvent(const DriverData& request) { // 1. Parse NBD request switch(request.type) { case DriverData::READ: { // 2. Create command auto cmd = std::make_shared<ReadCommand>( request.offset, request.length); // 3. Queue to ThreadPool thread_pool_.AddCommand(cmd); } break; case DriverData::WRITE: { auto cmd = std::make_shared<WriteCommand>( request.offset, request.buffer); thread_pool_.AddCommand(cmd); } break; } } private: ThreadPool& thread_pool_; };

3. ThreadPool Task Pattern

ThreadPool executes tasks (ICommand objects) from WPQ in priority order.

class ThreadPool { public: void AddCommand(std::shared_ptr<ICommand> cmd) { // Add to priority queue command_queue_.Push(cmd); // Wake up sleeping workers cv_.notify_one(); } private: void WorkerFunc() { while(running_) { // Wait for command auto cmd = command_queue_.Pop(); if (!cmd) continue; // Execute command try { cmd->Execute(); } catch (const std::exception& e) { logger_.Error("Command failed: %s", e.what()); } } } WPQ<std::shared_ptr<ICommand>> command_queue_; std::vector<std::thread> workers_; std::condition_variable cv_; };

🌐 Network Protocol Specification

Master-Minion Communication Protocol

All communication between master and minions uses UDP with custom binary protocol.

Message Format (Binary)

Master → Minion: REQUEST MESSAGE
┌─────────┬──────────┬──────────┬─────────────┬──────────┐
│ MSG_ID  │ Op Type  │ Offset   │ Length      │ Payload  │
│ (4B)    │ (1B)     │ (8B)     │ (4B)        │ (var)    │
└─────────┴──────────┴──────────┴─────────────┴──────────┘

Minion → Master: RESPONSE MESSAGE
┌─────────┬──────────┬──────────┬─────────┬──────────┐
│ MSG_ID  │ Status   │ Length   │ Reserved│ Data     │
│ (4B)    │ (1B)     │ (4B)     │ (4B)    │ (var)    │
└─────────┴──────────┴──────────┴─────────┴──────────┘

Op Type Values:
  0x01 = GET_BLOCK    (Read)
  0x02 = PUT_BLOCK    (Write)
  0x03 = DELETE_BLOCK (Trim)
  0x04 = PING         (Health check)

Status Values:
  0x00 = SUCCESS
  0x01 = ERROR_NOT_FOUND
  0x02 = ERROR_STORAGE_FULL
  0x03 = ERROR_CORRUPTED
  0xFF = ERROR_UNKNOWN

Message Examples

Example 1: READ Block 0 (1024 bytes) REQUEST: ┌────────┬────────┬────────────┬────────┐ │ 0x0042 │ 0x01 │ 0x00000000 │ 1024 │ │ (ID 66)│ GET │ Offset 0 │ Length │ └────────┴────────┴────────────┴────────┘ RESPONSE (Success): ┌────────┬────────┬────────┬─────────┬──────────────────┐ │ 0x0042 │ 0x00 │ 1024 │ 0x00000 │ [1024 bytes data]│ │ (ID 66)│ SUCCESS│ │ Reserved│ │ └────────┴────────┴────────┴─────────┴──────────────────┘ Example 2: WRITE Block 0 (1024 bytes) REQUEST: ┌────────┬────────┬────────────┬────────┬──────────────────┐ │ 0x0043 │ 0x02 │ 0x00000000 │ 1024 │ [1024 bytes data]│ │ (ID 67)│ PUT │ Offset 0 │ Length │ │ └────────┴────────┴────────────┴────────┴──────────────────┘ RESPONSE: ┌────────┬────────┬────────┬─────────┐ │ 0x0043 │ 0x00 │ 0 │ 0x00000 │ │ (ID 67)│ SUCCESS│ │ Reserved│ └────────┴────────┴────────┴─────────┘

Reliability Features

  • Message IDs: Track which request gets which response (handle out-of-order)
  • Timeouts: If no response in 5s, assume minion is dead
  • Retries: Exponential backoff: 1s, 2s, 4s, then give up
  • Checksums: Detect corrupted packets

🐕 Watchdog & Monitoring

Purpose of Watchdog

Minions might become unreachable due to:

  • Network failure (minion still alive, but network broken)
  • Power failure (minion crashed)
  • Software crash (minion process died)

Watchdog detects these failures and initiates recovery:

  • Mark minion as failed
  • Stop sending requests to dead minion
  • Use replicas (RAID01) for data access
  • Signal auto-discovery to find replacement minion

Watchdog Implementation

class Watchdog { private: std::thread monitor_thread_; std::map<int, MinionHealth> health_; std::mutex health_mutex_; public: void Start() { monitor_thread_ = std::thread([this]() { MonitorLoop(); }); } private: void MonitorLoop() { while (running_) { // Every 5 seconds, check each minion std::this_thread::sleep_for(5s); for (auto& [minion_id, health] : health_) { // If no response in last 15 seconds: if (now - health.last_response_time > 15s) { // Mark as failed health.status = MinionStatus::FAILED; // Notify RAID01Manager raid01_manager_.FailMinion(minion_id); // Notify auto-discovery to find replacement auto_discovery_.ReplaceMinion(minion_id); } } } } };

Health Check Mechanism

Timeline: Minion Health Monitoring

T=0s:    Watchdog: PING Minion 1
         │
         ├─ Minion 1 responds: PONG
         │  (Update: last_response = 0s, status = HEALTHY)

T=5s:    Watchdog: PING Minion 1
         │
         ├─ Minion 1 responds: PONG
         │  (Update: last_response = 5s, status = HEALTHY)

T=10s:   Watchdog: PING Minion 1
         │
         ├─ Minion 1: NO RESPONSE
         │  (Waiting for response... timeout after 2s)
         │
         ├─ Status: UNKNOWN (might be temporarily slow)

T=15s:   Watchdog: PING Minion 1
         │
         ├─ Minion 1: STILL NO RESPONSE
         │  (15s - 5s = 10s since last successful response)
         │
         ├─ Status: DEGRADED
         │  (Last response was at T=5s, but we tolerate up to 15s)

T=20s:   Watchdog: Check health
         │
         └─ Last response at T=5s
            Now: T=20s
            Elapsed: 15s (exceeds 15s threshold!)
            │
            ├─ Action: Mark FAILED
            ├─ RAID01 Manager: Minion 1 failed
            ├─ Scheduler: Stop retrying Minion 1
            └─ Auto-Discovery: Find replacement for Minion 1

🔎 Auto-Discovery Mechanism

Purpose

When a minion fails or new minions join, the system needs to dynamically discover and integrate them.

Auto-Discovery Flow

Scenario 1: New Minion Joins Network
──────────────────────────────────────

New Minion boots up
  │
  └─ Broadcasts: "Hello, I'm Minion-42, Port 9000"
     (UDP broadcast to network)
       │
       ▼
  DirMonitor hears broadcast
       │
       ├─ New minion discovered!
       │
       ├─ Add to RAID01Manager
       │  - minion_id = 42
       │  - ip = 192.168.1.100
       │  - port = 9000
       │
       ├─ Rebalance data
       │  - Identify blocks that need replica on new minion
       │  - Copy data from existing minion to new one
       │  - Now: Block X on MinionsA+B → MinionsB+42
       │
       └─ Update RAID01 mapping
          - Save new mapping to disk
          - Log: "Minion 42 joined, rebalanced 500MB data"

Scenario 2: Minion Dies, Auto-Discovery Replacement
─────────────────────────────────────────────────────

Watchdog: Minion 1 FAILED
  │
  ├─ RAID01 Manager marks Minion 1 as dead
  │
  ├─ For all blocks on Minion 1:
  │  └─ Find replica on other minion (e.g., Minion 2)
  │     (Data is safe, we have replica!)
  │
  ├─ Auto-Discovery: Find replacement
  │  └─ Broadcast: "Need backup for Minion 1 blocks"
  │
  ├─ New minion joins or rejoins network
  │  └─ Auto-Discovery identifies it
  │
  ├─ Copy data to new minion
  │  └─ All blocks from Minion 1 → replicate to new minion
  │
  └─ Update mapping
     └─ Block X: (Minion 2, New Minion) ✓

Auto-Discovery Implementation Sketch

class AutoDiscovery { private: void DiscoveryLoop() { while (running_) { // Listen for minion broadcasts MinionInfo discovered = ReceiveBroadcast(); if (!discovered) { continue; } // Check if minion is already known if (IsKnownMinion(discovered.id)) { // Minion rejoined after failure OnMinionRejoin(discovered); } else { // New minion discovered OnNewMinion(discovered); } } } void OnNewMinion(const MinionInfo& info) { // 1. Register with RAID01 Manager raid01_manager_.AddMinion(info); // 2. Start rebalancing RebalanceData(info.id); } void OnMinionRejoin(const MinionInfo& info) { // 1. Mark as healthy again watchdog_.ClearFailure(info.id); // 2. Resync data that might have changed ResyncData(info.id); } };

ReadCommand Implementation

void ReadCommand::Execute() { // 1. Find where block is stored (RAID01) auto [minion1, minion2] = raid01_manager_.GetBlockLocation(offset_); // 2. Try primary minion first uint32_t msg_id = minion_proxy_.SendGetBlock(minion1, offset_, length_); // 3. Wait for response (async, with timeout) auto response = scheduler_.WaitForResponse(msg_id, 5000ms); if (response.status == SUCCESS) { // 4a. Got data! Return to user SendReply(response.data); return; } // 4b. Primary minion failed, try replica msg_id = minion_proxy_.SendGetBlock(minion2, offset_, length_); response = scheduler_.WaitForResponse(msg_id, 5000ms); if (response.status == SUCCESS) { SendReply(response.data); return; } // Both minions failed - data loss! SendError("RAID01 failure: both replicas unreachable"); }

WriteCommand Implementation

void WriteCommand::Execute() { // 1. Find where block should be stored (RAID01) auto [minion1, minion2] = raid01_manager_.GetBlockLocation(offset_); // 2. Send to both minions (RAID01 = 2 copies) uint32_t msg_id1 = minion_proxy_.SendPutBlock(minion1, offset_, data_); uint32_t msg_id2 = minion_proxy_.SendPutBlock(minion2, offset_, data_); // 3. Wait for both responses auto resp1 = scheduler_.WaitForResponse(msg_id1, 5000ms); auto resp2 = scheduler_.WaitForResponse(msg_id2, 5000ms); // 4. Check if at least one succeeded (RAID01 requirement) if (resp1.status == SUCCESS || resp2.status == SUCCESS) { SendSuccess(); return; } // Both copies failed - write failed SendError("RAID01: failed to write 2 copies"); }

Scheduler with Exponential Backoff

Response Scheduler::WaitForResponse(uint32_t msg_id, std::chrono::milliseconds timeout) { auto start = std::chrono::now(); int retry_count = 0; int retry_delay = 100; // Start at 100ms while (true) { // 1. Wait for response auto response = response_manager_.GetResponse(msg_id); if (response) { return response; } // 2. Check timeout auto elapsed = std::chrono::now() - start; if (elapsed > timeout) { return Response{ERROR, "Timeout"}; } // 3. Retry with exponential backoff std::this_thread::sleep_for(std::chrono::milliseconds(retry_delay)); retry_delay = std::min(retry_delay * 2, 5000); // Cap at 5s retry_count++; } }

RAID01 Manager Block Mapping

std::pair<int, int> RAID01Manager::GetBlockLocation(uint64_t block_num) { // Simple RAID01 mapping strategy: // Block N goes to minions: (N % num_minions) and ((N+1) % num_minions) int total = minions_.size(); int minion1 = block_num % total; int minion2 = (block_num + 1) % total; // Handle failed minions - try to find healthy ones while (minions_[minion1].status == FAILED) { minion1 = (minion1 + 1) % total; } while (minions_[minion2].status == FAILED || minion2 == minion1) { minion2 = (minion2 + 1) % total; } return {minion1, minion2}; }

Common Questions for Newcomers

Q: Why do we need both Reactor and ThreadPool?
A: Reactor is for I/O (fast, non-blocking). ThreadPool is for computation (can be slow, CPU-bound). Keep them separate so fast I/O isn't blocked by slow computation.
Q: Why is RAID01 better than just storing one copy?
A: Fault tolerance. If a minion dies, we still have the data. With one copy, data loss = data gone forever. RAID01 survives 1 minion death.
Q: Why can't we just wait for minion responses?
A: Network is slow. If master waits for minion, master blocks. While waiting, can't process other requests. Async handles multiple requests concurrently.
Q: How do plugins work?
A: Plugin is a .so file. When loaded, it calls Factory.Add("NEW_CMD", creator_func). Now Factory can create new command types. No recompile needed.

📚 Glossary

Term Definition
Admin Command High-priority command that must execute immediately (e.g., FLUSH, shutdown)
Async Asynchronous - not waiting for completion. Request is sent, handler continues
Block A unit of data storage. LDS divides files into blocks for RAID01
CPF Command Processing Framework - the core of LDS that handles I/O
Concurrent Multiple things happening at overlapping times (not necessarily simultaneously)
Determinism Behavior is predictable and consistent. Same input always gives same output
Epoll Linux system call for efficient I/O multiplexing. Scales to thousands of connections
Factory Design pattern for creating objects without specifying exact classes
Fault Tolerance System continues working even when some components fail
GI / GO Generic Input / Generic Output - interfaces for CPF communication
Handler Function that responds to an event (e.g., Reactor calls handler when data arrives)
IoT Internet of Things - small networked devices (e.g., Raspberry Pi)
Master Central LDS server that coordinates all operations
Minion IoT device (Raspberry Pi) that stores data blocks
Mutex Mutual exclusion lock - prevents multiple threads from accessing data simultaneously
NBD Network Block Device - Linux kernel module that creates virtual block device
Non-blocking Operation returns immediately without waiting for completion
Parallel Multiple things happening simultaneously (requires multiple cores)
Plugin Loadable module (.so file) that extends system functionality without recompiling
Priority Queue Queue where elements are ordered by priority (high priority items pop first)
RAID01 Data replication scheme - each block stored on 2 minions for fault tolerance
Reactor Design pattern for handling I/O events efficiently (using epoll)
Replica Copy of data. RAID01 keeps 2 replicas of each block
Response Manager Component that listens for async responses from minions
Scheduler Component that retries failed operations and manages delayed execution
Thread Lightweight process. Multiple threads share same memory but run independently
Thread-safe Can be safely used by multiple threads without corrupting data
ThreadPool Collection of worker threads that execute tasks from a queue
Timeout Maximum time to wait for something. If timeout expires, give up and retry
UDP User Datagram Protocol - fast, unreliable network protocol (unlike TCP)
Waitable Threads can wait on it. When data arrives, waiting threads wake up
WPQ Waitable Priority Queue - thread-safe queue with priority ordering
